Version Control and Git

June 15, 2015

What is version control?

Version control is a system for keeping track of individual changes to a file over time. You can use version control to see who made what change and when. Each change is usually accompanied by a comment which explains the purpose behind the change, so you can also understand why a given change was made.

Version control is a concept that can be useful for a variety of disciplines, not just software development. A book author, for example, can track revisions to his work. If she is not happy with the latest revision of a chapter, she can simply revert back to an earlier version. At the very least, version control will help protect a user from losing his work.

For software deveopment in particular, version control has many useful applications. First, it can allow multiple developers to work on the same codebase without trampling over each other's changes. Second, version control can even help an invidual programmer to better organize his work and rollback his mistakes. For me personally, I know that most of my decisions are likely to be wrong. Therefore, I like to have at most one decision per commit, which will allow me to most easily roll back decisions that turn out to be wrong. If I instead lump all of my decisions together into one giant document without version control, I would then have a bunch of wrong decisions lumped together and it may be difficult to tease them apart later.

What is Git?

Git is a version control system (VCS) which offers the functions described above. Git functions as a distributed file system, which means that users will have their own working copy of the entire repository on their local machine. They will be making commits to their local copy of the repository and will have to explicitly push their changes to the remote repository. The advantage of this approach is that every local copy of the repository can effectively serve as a backup for the central remote repository if it were to crash.

What makes Git unique compared to other VCS examples is the way it thinks about and stores data. Most VCS's represent the data as a series of changes to its files. So, if I were to add a line to file1, a typical VCS would store my change to file1. Git, however, does not take this approach. Instead, Git represents a commit as a snapshot of all the files in my database at the time of my commit. You can think of it as saving a Word document. When you save a Word document, you are capturing the state of that document at the precise moment you hit the save button.

What is GitHub?

GitHub serves as a "hub" for storing Git repositories. As mentioned above, developers explicitly push their local commits to a central remote repository. GitHub serves as a convenient place to store the remote repository. The advantage of using Github is that now multiple developers from various parts of the world can collaborate. They can each check out a local copy of the central GitHub repository, contribute their changes to the codebase and then push back to GitHub. Github will merge everyone's changes into one unified whole.
